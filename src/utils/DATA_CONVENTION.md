# 🏗 AR15 BIM Data Convention (v2.0)
คู่มือมาตรฐานการจัดการข้อมูลสินทรัพย์อาคาร AR15 สำหรับระบบ Knowledge Graph และ 3D Digital Twin

---

## 📂 1. โครงสร้างการจัดเก็บไฟล์ (File Structure)
**กฎเหล็ก:** ห้ามแก้ไขไฟล์ `src/utils/AR15-DATA.md` โดยตรง เพราะเป็นไฟล์ที่ถูกสร้างขึ้นโดยอัตโนมัติ (Generated) ให้แก้ไขไฟล์แยกตามหมวดหมู่ในโฟลเดอร์ `src/utils/data/` เท่านั้น:

*   **`ar15-common.md`**: ข้อมูลโครงสร้างหลักอาคาร (Root), อุปกรณ์ส่วนกลาง (CU, LP) และระบบ CCTV
*   **`ar15-floor-1.md`**: รายละเอียดห้องและอุปกรณ์ทั้งหมดในชั้น 1 (AC, EE, FUR, ARCH)
*   **`ar15-floor-2.md`**: รายละเอียดห้องและอุปกรณ์ทั้งหมดในชั้น 2 (AC, EE, FUR, ARCH)

## 📏 2. กฎการเขียนข้อมูล (Syntax Rules)

### **2.1 การย่อหน้า ( TAB - Indentation)**
*   **ต้องใช้ 4 Spaces เท่านั้น** ในการแบ่งระดับชั้น (Hierarchy)
*   ใช้ Tab ที่ตั้งค่าให้ 4 space ห้ามใช้ จำนวน Space อื่นๆ เพราะจะทำให้ระบบ `data:check` ตรวจพบและแสดงข้อผิดพลาด

### **2.2 การนิยามโหนด (Node Definition)**
*   ใช้เครื่องหมาย `-` นำหน้าชื่อโหนดเสมอ
*   **รูปแบบ:** `- [ชื่อโหนด] { [ข้อมูล Metadata] }`
*   **ตัวอย่าง:** `- FCU-101-1 { Type: 42TGF0361CP, AssetID: สถ.2568-xxx }`

### **2.3 ข้อมูลคุณสมบัติ (Metadata Blocks)**
*   ต้องครอบด้วยเครื่องหมายปีกกา `{ }`
*   สามารถเขียนแบบบรรทัดเดียว หรือหลายบรรทัดก็ได้ (Multi-line)
*   **กฎเหล็ก:** ต้องเปิด `{` และปิด `}` ให้ครบถ้วนเสมอ

## 🏛 3. ลำดับชั้นข้อมูลและการเชื่อมโยง 3D (Hierarchy & 3D Mapping)
ชื่อโหนดทุกตัวต้อง **ตรงกันทุกตัวอักษร** กับชื่อ Object ในไฟล์ `.glb` และต้องเป็น **Unique Name** ทั่วทั้งอาคาร:

### **3.1 โครงสร้างลำดับชั้น (Standard Hierarchy)**
```text
- ar15 (Building Node)
    - floor-x (Floor Node)
        - room-xxx (Room Node)
            - [Room-ID]-AC / EE / FUR / ARCH (System Headers - Auto Prefixed)
                - AC-xxx (Asset Set Group)
                    - FCU-xxx (Physical Object in 3D)
                    - CDU-xxx (Physical Object in 3D)
```

### **3.2 กลไกชื่อไม่ซ้ำอัตโนมัติ (Auto-Prefix Logic)**
เพื่อให้การเขียนใน Markdown ทำได้ง่าย แต่ยังคงความเป็น Unique ในระดับสากล:
*   **Generic Headers:** ชื่อกลุ่มระบบ (`AC`, `EE`, `ARCH`, `FUR`, `CCTV`) ระบบจะเติมชื่อ Parent (เช่น ชื่อห้อง) ไว้ข้างหน้าให้อัตโนมัติในฐานข้อมูล (เช่น `room-101-AC`)
*   **Specific Assets:** ชื่ออุปกรณ์จริง (`FCU-xxx`, `CDU-xxx`, `LP-xxx`) **ต้องตั้งชื่อให้ไม่ซ้ำกันเลยตั้งแต่ต้น** ห้ามซ้ำกันข้ามห้องหรือข้ามชั้นเด็ดขาด

### **3.3 การจับคู่กับ 3D Model (3D Matching)**
*   ทุกโหนดที่มีชื่อปรากฏหลังเครื่องหมาย `-` ระบบจะถือว่ามีตัวตนใน 3D
*   ชื่อใน Markdown == ชื่อ Object (หรือ GlobalId) ในไฟล์ `.glb`
*   หากชื่อไม่ตรงกัน ระบบ Graph จะไม่สามารถสั่งงานหรือ Highlight วัตถุใน 3D ได้

## 🛠 4. ขั้นตอนการทำงาน (Workflow)
ทุกครั้งที่มีการแก้ไขข้อมูล ให้รันคำสั่งตามลำดับดังนี้:

1.  **แก้ไขไฟล์:** แก้ที่ `src/utils/data/ar15-xxx.md`
2.  **ตรวจสอบ (Validate):** 
    ```bash
    npm run data:check
    ```
    *(เพื่อดูว่ามี Error เรื่อง Indent, วงเล็บปีกกา หรือชื่อโหนดซ้ำหรือไม่)*
3.  **รวมและแปลงไฟล์ (Generate):** 
    ```bash
    npm run data:generate
    ```
    *(รวมไฟล์ย่อย -> สร้าง AR15-DATA.md -> แปลงเป็น AR15.json)*
4.  **ส่งขึ้นระบบ (Sync - ถ้าจำเป็น):** 
    ```bash
    npm run data:sync
    ```
    *(ส่งข้อมูลเข้าสู่ Knowledge Graph บน Supabase)*

## 🔍 5. กฎการตรวจสอบ (Validation Logic)
คำสั่ง `npm run data:check` จะตรวจสอบสิ่งเหล่านี้โดยอัตโนมัติ:
1.  **Indent Check:** บรรทัดที่ขึ้นต้นด้วย `-` ต้องเยื้องเป็นทวีคูณของ 4 เท่านั้น
2.  **Brace Balance:** ตรวจสอบปีกกา `{ }` ว่าเปิด-ปิดคู่กันครบไหม (รองรับการเขียนหลายบรรทัด)
3.  **Duplicate Name Check:** ตรวจสอบชื่อโหนดห้ามซ้ำกัน (Global Unique) ยกเว้นกลุ่ม Generic Headers ที่ระบบอนุญาตให้ซ้ำได้เพราะจะมีการเติม Prefix ให้ภายหลัง

## 🎨 6. สถานะและสี (Status & Keywords)
ระบบจะตรวจจับคำสำคัญในฟิลด์ `status` เพื่อเปลี่ยนสีใน 3D และ UI:

| สถานะ | ตัวอย่าง Keyword | สีใน 3D Graph |
| :--- | :--- | :--- |
| **ปกติ** | `Normal`, `Completed`, `ปกติ` | 🟢 เขียว / ฟ้า |
| **รอซ่อม** | `Maintenance`, `Warning`, `Pending`, `ซ่อม` | 🟡 ส้ม |
| **ชำรุด** | `Faulty`, `Broken`, `พัง`, `เสีย` | 🔴 แดง |
