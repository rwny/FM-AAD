# หลักการบันทึกข้อมูลอาคาร (BIM/FM Data Convention)
คู่มือการบันทึกข้อมูลด้วยรูปแบบ Markdown เพื่อแปลงเป็น JSON สำหรับใช้งานในระบบ 3D/AR

## 1. โครงสร้างลำดับชั้น (Indentation Rule)
การเว้นวรรค (Indent) คือหัวใจสำคัญในการระบุความสัมพันธ์แบบ "Parent-Child" (เก็บในตาราง `kg_edges` เป็นความสัมพันธ์แบบ `contains`) **ต้องใช้ 4 spaces ต่อ 1 ระดับ**

- **LEVEL 0:** อาคาร (เช่น `- AR15`)
    - **LEVEL 1:** ชั้น (เช่น `- FLOOR 1`)
        - **LEVEL 2:** ห้อง (เช่น `- rm-101 : ชื่อห้อง`)
            - **LEVEL 3:** ระบบงาน (เช่น `- AC`, `- EE`, `- FUR`)
                - **LEVEL 4:** อุปกรณ์ (เช่น `- INSTANCE-ID { ... }`)

**หมายเหตุ:** LEVEL 4 คือระดับสุดท้ายใน Markdown 

**ข้อสังเกต:** ข้อมูลที่เป็น Attribute (คุณสมบัติ) จะเขียนต่อท้ายชื่อโหนดใน `{ ... }` ไม่ถือเป็นการเพิ่มระดับ Indent ใหม่

---

## 2. การนิยามประเภท (Asset Type Definition)
ใช้สำหรับเก็บข้อมูลส่วนกลางที่ทรัพย์สินรุ่นเดียวกันมีเหมือนกัน เพื่อไม่ต้องเขียนซ้ำ
- **รูปแบบ:** `- [ID-NAME] { Key: Value, Key: Value }`
- **ตัวอย่าง:**
    ```markdown
    - [LF-1] { Type: โต๊ะ 4 ขา, Brand: ModernForm, Model: MD-1565465 }
    ```

---

## 3. การบันทึกทรัพย์สินรายชิ้น (Asset Instance)
คือของที่มีอยู่จริงในห้องนั้นๆ โดยอ้างอิง ID มาจากประเภทข้างต้น
- **รูปแบบ:** `- ID.ลำดับ { AssetID: xxx, Install: yyyy-mm-dd }`
- **ตัวอย่าง:**
    ```markdown
    - LF-1.1 { AssetID: สถ.2568-1109, Install: 2026-04-01 }
    ```

---

## 4. กฎพารามิเตอร์และโหนดลูก (Attributes vs Child Nodes)
- **Attributes:** ข้อมูลคุณลักษณะ (Metadata) ต้องเขียนอยู่ในเครื่องหมายปีกกา `{ ... }` ในบรรทัดเดียวกันเท่านั้น
- **Child Nodes:** ทุกบรรทัดที่เยื้อง (Indent) เข้าไปใต้โหนดแม่ จะถูกถือว่าเป็น **Node** ลูกในตาราง `kg_edges` เสมอ (ความสัมพันธ์แบบ `contains`) ไม่ใช่ attribute ของโหนดแม่

---

## 5. การจัดการประวัติการบริการ (Service Log)
เพื่อให้ข้อมูลใน Markdown ไม่ซับซ้อนเกินไป ข้อมูลประวัติการซ่อมบำรุงทั้งหมด (Service Log) จะถูกแยกไปจัดการในระบบฐานข้อมูล **Supabase** แทน
- **ตารางที่ใช้:** `maintenance_logs`
- **การเชื่อมโยง:** ใช้ `asset_id` (UUID) จากตาราง `assets` ในการอ้างอิง
- **โครงสร้างข้อมูลใน Service Log:**
    - **Date:** วันที่เข้าดำเนินการ (ISO Format)
    - **Issue:** รายละเอียดปัญหาที่พบหรือหัวข้อการตรวจเช็ค
    - **Reporter:** ชื่อผู้แจ้งซ่อมหรือผู้รับผิดชอบภายใน
    - **Contractor:** ชื่อบริษัทคู่สัญญาหรือช่างภายนอก (เช่น 'ModernForm Service Team')
    - **Status:** สถานะปัจจุบัน (`Pending`, `In Progress`, `Completed`)
    - **Note:** หมายเหตุเพิ่มเติมหรือรายละเอียดการแก้ไข
- **จุดเด่น:** ระบบจะเก็บประวัติแยกเป็นรายชิ้น (Asset-by-asset) ทำให้สามารถคำนวณสถิติการเสีย (MTBF) และวางแผนซ่อมบำรุงเชิงป้องกันได้แม่นยำกว่าการจดใน Text file

---

## 5. ข้อควรระวัง (Do's & Don'ts)
- ✅ **Do:** ใช้เครื่องหมาย `:` คั่นระหว่าง Key และ Value โดยมีช่องว่างเสมอ (เช่น `Key : Value`)
- ✅ **Do:** รักษาระดับ Indent ให้ตรงกันตลอดทั้งไฟล์
- ❌ **Don't:** ห้ามใช้ Tab (ให้ใช้ Space 4 ครั้งแทน) เพื่อป้องกันความเพี้ยนในบาง Editor
- ❌ **Don't:** หลีกเลี่ยงการใช้เครื่องหมายพิเศษอื่นๆ ในชื่อ ID นอกจาก `-` และ `.`

---

## 6. การจัดการชื่อโหนดซ้ำ (Unique Naming Convention)
เพื่อให้ระบบฐานข้อมูล Knowledge Graph สามารถแยกแยะทรัพย์สินในแต่ละห้องได้ถูกต้อง โดยที่ผู้เขียน Markdown ยังคงความเรียบง่ายไว้ ระบบจะมีการจัดการชื่ออัตโนมัติ ดังนี้:

### หลักการ Prefix สำหรับหมวดหมู่ระบบ (System Groups)
โหนดที่เป็นชื่อหมวดหมู่ทั่วไป เช่น **`AC`**, **`EE`**, **`ARCH`**, **`FUR`** จะถูกเติมชื่อ Parent (ชื่อห้อง) ไว้ข้างหน้าโดยอัตโนมัติในขั้นตอน Sync เพื่อป้องกันข้อมูลทับซ้อนกัน

*   **ใน Markdown เขียน:**
    ```markdown
    - room-101
        - AC
    - room-102
        - AC
    ```
*   **ใน Database (Backend) จะเก็บเป็น:**
    *   `room-101-AC`
    *   `room-102-AC`
*   **การแสดงผล (Frontend):**
    *   ระบบจะใช้ค่าจาก `metadata.display_name` ซึ่งเก็บคำว่า **"AC"** ไว้ ทำให้บนกราฟ 3D หรือ UI ยังคงเห็นเป็นคำสั้นๆ ว่า **"AC"** ตามเดิม

**หมายเหตุ:** ผู้บันทึกข้อมูลใน Markdown ไม่ต้องเติม Prefix เอง ให้เขียนชื่อสั้นๆ ตามปกติ สคริปต์ `sync-ac-to-supabase.cjs` จะจัดการส่วนนี้ให้เองครับ

```markdown
- AR15
    - FLOOR 1
        - rm-101 : ห้องประชุม
            - FUR
                - LF
                    - Table-1-1 { Type:IKEA5485, AssetID: สถ.2568-table-001, InstallDate:YYYY-MM-DD }
```

### 2.4 คุณสมบัติพิเศษ (Special Properties)
- **ConnectsFrom / ConnectsTo**: ใช้ระบุทิศทางการไหลของพลังงานหรือสาร (เช่น สายไฟ, ท่อน้ำ)
- **Monitors**: ใช้สำหรับกล้อง CCTV เพื่อเชื่อมโยงกับห้องที่ทำการบันทึกภาพ
- **Type / AssetID / InstallDate**: ข้อมูลพื้นฐานสำหรับคำนวณอายุ (Age) และแสดงผลในการ์ดข้อมูล
